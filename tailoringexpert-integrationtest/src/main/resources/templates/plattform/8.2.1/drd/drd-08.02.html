DRD-08.02 - Software Development Plan
Reference: ECSS-E-ST-40C Space engineering - Software
ECSS-Q-ST-80C Space product assurance - Software product assurance
Purpose: The Software Development Plan describes the established management and development approach for the software items that a software supplier has to define to set up a software project in accordance with the customer requirements.
Content: Title
  1. The Software Development Plan to be created on this DRD shall be titled "[Insert a descriptive modifier] Software Development Plan".
  2. The descriptive modifier shall be selected to clearly identify the applicable product and level specification.
Title page
The title page shall identify the project software test plan identification number, title of the test plan, date of release and release authority.

Contents list
The content list shall identify the title and location of every clause and major subclause, figure, table and annex contained in the software test plan.

Foreword
A foreword shall be included which describes as many of the following items as appropriate:
  • identification of which organizational entities prepared the Software Development Plan;
  • information regarding the approval of the Software Development Plan;
  • identification of other organizations that contributed to the preparation of the Software Development Plan;
  • a statement of effectivity identifying which other documents are cancelled and replaced in whole or in part;
  • a statement of significant technical differences between this document and any previous document;
  • the relationship of the Software Development Plan to other standards or documents.

Introduction

An introduction may be included to provide specific information or commentary about the technical contents of the Software Development Plan.

4 Project organization

This clause shall be organized in subclauses addressing the software process model, the organizational structure, the organizational boundaries and interfaces and the project responsibilities.

4.1 Software process model

  1. This subclause shall define the activities to be performed in the project and their inputs and outputs.
  2. The definition shall include the major project functions (i.e. activities that span the entire duration of the project, such as project management, configuration management, verification and validation, and quality assurance) and the major production activities needed to achieve the project objectives.
  3. The process model shall be based on the ECSS software standards

    The definition of the process model may be textual or graphic

  4. any related tailoring shall be justified and documented.

4.2 Organizational structure

  1. This subclause shall describe the organizational structure to be used in the project.

    Lines of reporting, control and communication can be described using organigrams.

  2. The roles identified in the organizational structure shall also be identified, describing the purpose of the role and listing the associated responsibilities. As an example, roles that often appear in the software project organizational structure are:
  • software manager;
  • software engineers;
  • software configuration manager;
  • software product assurance engineer.
  1. This subclause shall also provide references to the plans relevant to the project support functions: the software configuration management plan, the software verification plan, the software validation plan and the software product assurance plan.

4.3 Organizational boundaries and interfaces

This subclause shall describe the relationship between all the involved parties and the responsibility of each organization/team according to the established customer/supplier relationship.

For example, the relationship and reporting lines between customer/supplier/maintainer can be addressed.

5 Software project management approach

This clause shall be organized in subclauses addressing the management objectives, assumptions, dependencies and constraints, risk management, monitoring and controlling mechanisms, staffing plan, supplier management and software acquisition process.

5.1 Management objectives and priorities

  1. This subclause shall define the management objectives of the software project and associated priorities.
  2. Objectives shall be discussed and trade-offs documented for their prioritization.

5.2 Assumptions, dependencies and constraints

This subclause shall state:
  • the assumptions on which the plan is based;
  • the external events the project is dependent upon;
  • constraints on the project;
  • technical issues.

    Technical issues can only be mentioned if they have an effect on the plan. Assumptions, dependencies and constraints are often difficult to distinguish. The best approach is not to categorize them but to list them.

For example:
  • limitations on the budget;
  • schedule constraints (e.g. launch dates, delivery dates);
  • constraints on the location of staff (e.g. they must work at developer's premises);
  • commercial hardware or software that will be used by the system;
  • availability of simulators and others test devices;
  • availability of external systems with which the system interfaces.

5.3 Risk management

This subclause shall describe the contribution of the software engineering function to the project risk management approach.

5.4 Monitoring mechanisms

  1. This subclause shall define the monitoring mechanisms for managing the work (e.g. progress report, progress meeting, action item lists).
  2. This shall apply to both the customer relationships and the contractor's relationships.

5.5 Staffing plan

This subclause shall specify the roles and skills of staff that will be involved in the project.

5.6 Software procurement process

The SDP shall describe the software procurement process implementation, and include here or in annex the procured software component list.

6 Software development approach

This clause shall be organized in subclauses addressing the overall strategy to the software development, the software development life cycle, the software engineering standards and techniques, the software development and testing environment, the software documentation plan, the project support functions.

6.1 Strategy to the software development

This subclause shall address the overall strategy to the software development.

An activity diagram can be included.

6.2 Software project development life cycle

This subclause shall be organized in subclause addressing the identification of the software development life cycle, the definition of milestones and associated documentation to be produced.

6.2.1 Software development life cycleidentification

  1. This subclause shall define the software development life cycle.
  2. Definition of the selected life cycle paradigm (e.g. waterfall, incremental, evolutionary) as well as the adopted software versioning approach shall be identified.
  3. This subclause shall cover the implementation of all the activities/tasks relevant to the involved software processes:
  • system engineering processes related to software;
  • software requirement engineering process;
  • software design engineering process;
  • software validation with respect to the requirements baseline (RB);
  • software verification and validation processes;
  • software operation process;
  • software maintenance process;
  • software management process;
  • software problem/change management process.

6.2.2 Relationship with the system development life cycle

This subclause shall describe the phasing of the software life cycle to the system development life cycle.

A process model representation can be used.

6.2.3 Reviews identification and associated documentation

This subclause shall describe scope and purpose of each identified review, relevant deliverables and expected outputs.

6.3 Software engineering standards and techniques

  1. This subclause shall describe (or provide references to their description of) the applied methodologies and define the standards for each software process and relevant activity.
  2. The requirements analysis method used shall be identified and referenced.

    Reference to applied literature or other standards can be described here

  3. Any adaptation of the requirements analysis method shall be defined or referenced
  4. The selected design (architectural design and detailed design) methods shall be identified and referenced.

    Reference to applied literature or other standards can be described here.

  5. The specificity of automatic code generation tool chains shall be considered, in particular for maintenance.
  6. Any adaptation of the design method (e.g. deviations, extensions, and avoidance of utilization of some methodology features...) shall be defined or referenced.
  7. Any required verification/validation method to conduct verification and validation testing activities shall be identified. For example, the definition of any test procedure language for software validation has to be specified here.
  8. Any MMI standard to be applied to the software development, if code generators are utilized (e.g. constraints to be imposed to use of generators in terms of allowed specific features) shall be documented.
  9. The selected software delivery format shall be defined.

6.4 Software development and software testing environment

  1. This subclause shall describe the software development environment and testing environment, including the evidence of its suitability and the programming language selection suitability.
  2. Hardware platforms and selected software tools to be utilized for the software development and testing shall be defined and described with justification of their selection with respect to the relevant software methodology and standards.

    This covers, in particular, the tools used to configure the software for a particular mission with the parameters of the mission database.

  3. This subclause shall, then, identify, for example:
  • the compiler and cross compiler system;
  • the requirements analysis tool;
  • the tools utilized in the software image generation;
  • the configuration management tools;
  • the software design tools;
  • the software static analysis tools
  • the software test scripts language tools;
  • the software testing tools (debuggers, in circuit emulator, bus analyzer).

6.5 Software documentation plan

This subclause shall define or refer to all documentation relevant to the project and the documentation standards applied to the software project. The subclause should be then structured as follows:

6.5.1 Software documentation identification

This subclause shall, for each document to be produced (both internal documents and deliverables), include the documentation plan specifying:
  • the documentation file;
  • the document name;
  • the delivery requirements;
  • the review requirements;
  • the approval requirements.

6.5.2 Software documentation standards

  1. This subclause shall describe the documentation standards applicable to the project.
  2. Any tailoring to applicable documentation standards shall be specified in this subclause.

    Depending on the specific project documentation requirements, this subclause can either refer to applicable documentation standards for format and layout (for example to the DRD contained in the ECSS-E-40) or specify the proposed documentation templates.

6.6 ECSS standard tailoring traceability

This subclause shall include the coverage matrix of the applicable tailoring of ECSS-E-ST-40, or provide a reference to it.
