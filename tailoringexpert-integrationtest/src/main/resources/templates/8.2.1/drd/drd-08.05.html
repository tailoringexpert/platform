DRD-08.05 - Software [Unit / Integration] Test Plan
Reference: ECSS-E-ST-40C Space engineering - Software
ECSS-Q-ST-80C Space product assurance - Software product assurance
Purpose: To provide rules and information for the production of test plans for the unit test plan and the integration test plan.
Content:

Title

The Software [Unit / Integration] Test Plan to be created on this DRD shall be titled as one of the following:
  • [Project] Software Unit Test Plan,
  • [Project] Software Integration Test Plan

Title page

The title page shall identify the project Software [Unit / Integration] Test Plan identification number, title of the test plan, date of release and release authority.

Contents list

The content list shall identify the title and location of every clause and major subclause, figure, table and annex contained in the Software [Unit / Integration] Test Plan.

Foreword

A foreword shall be included which describes as many of the following items:
  • identification of which organizational entities prepared the Software [Unit / Integration] Test Plan;
  • information regarding the approval of the Software [Unit / Integration] Test Plan;
  • identification of other organizations that contributed to the preparation of the Software [Unit / Integration] Test Plan;
  • a statement of effectivity identifying which other documents are cancelled and replaced in whole or in part;
  • a statement of significant technical differences between this document and any previous document;
  • the relationship of the Software [Unit / Integration] Test Plan to other standards or documents.

Introduction

An introduction can be included to provide specific information or commentary about the technical contents of the Software [Unit / Integration] Test Plan.

1 Scope and applicability

This clause shall describe the scope, applicability and purpose of the Software [Unit / Integration] Test Plan.

1.1 Scope

This subclause shall contain a statement to define the goal of the Software [Unit / Integration] Test Plan.

1.2 Purpose

This subclause shall contain a statement to define the use of the Software [Unit / Integration] Test Plan.

4 Software overview

This subclause shall contain a brief description of the software under test and its context: a summary its configuration, its operational environment and its external interfaces. Reference to technical of its functionality, documentation can be done.

5 Software unit testing / software integration testing

This clause shall describe responsibility and schedule information for the software unit testing / integration testing. It shall contain the following subclauses:

5.1 Organization

This subclause shall describe the organization of software unit testing / integration testing activities. Topics that should be included are:
  • roles,
  • reporting channels,
  • levels of authority for resolving problems,
  • relationships to the other activities such as project management, development, configuration management and product assurance.

5.2 Master schedule

This subclause shall identify the schedule for the software unit testing / integration testing activities, in particular, test milestones identified in the software project schedule and all item delivery events. This subclause should specify:
  • any additional test milestones and state the time required for each testing task,
  • the schedule for each testing task and test milestone,
  • the period of use for the test facilities.

5.3 Resource summary

This subclause shall summarize the resources needed to perform the software unit testing / integration testing activities such as staff, hardware and software tools.

5.4 Responsibilities

This subclause shall define the specific responsibilities associated with the roles described in subclause 5.1. In particular, this subclause should identify the groups responsible for managing, designing, preparing, executing the tests. Groups may include developers, technical support staff, and product assurance staff.

5.5 Tools, techniques and methods

This subclause shall identify the hardware platforms, software tools, techniques and methods used for software unit testing / integration testing activities.

5.6 Personnel and personnel training requirements

This subclause shall list any requirement for software unit testing / integration testing personnel and any necessary training needs.

5.7 Risks and contingencies

This subclause shall identify risks to the software unit testing / integration testing campaign. Contingency plans should be specified.

6 Control procedures for software unit testing / integration testing

This clause shall contain information (or reference to) about applicable management procedures concerning the following aspects:
  • problem reporting and resolution;
  • deviation/waiver policy;
  • control procedures.

7 Software unit testing / integration testing task identification

This clause shall describe the approach to be utilized for the software unit testing / integration testing. It shall describe the software integration strategy. It shall specify which are the tasks and the items under tests, as well as criteria to be utilized. It shall contain the following subclauses:

7.1 Features to be tested

This subclause shall identify all the features to be tested, making references to the applicable documentation.

7.2 Features not to be tested

This subclause shall identify all the features and significant combinations not to be tested.

7.3 Test pass / fail criteria

This subclause shall specify the general criteria to be used to determine whether or not test are passed.

7.4 Manually and automatically generated code

This subclause shall address separately the activities to be performed for manually and automatically generated code, although they have the same objective.

8 Software unit test / integration test design

This clause shall provide for definition of unit / integration test design. It shall be organized per identified test design. This subclause may be simplified in the software unit test plan.

8.n Test design "n"

This subclause shall provide for definition of the unit test / integration test design "n". For the unit / integration test design "n", it shall contain the following subclauses:

8.n.1 Test design identifier

This subclause shall specify the test design uniquely. It shall briefly describe the test design.

8.n.2 Features to be tested

This subclause shall identify the test items and describe the features to be tested. Reference to appropriate documentation shall be performed and traceability information shall be provided.

8.n.3 Approach refinements

This subclause shall identify the test approach implemented for the specific test design and the specific test class implemented. This description shall provide the rationale for the test case selection and grouping into test procedures. The method for analyzing test results shall be identified (e.g. compare with expected output, compare with old results, ...). Configuration of the facility (both hardware and software) required to execute the identified test shall be described.

8.n.4 Test case identification

This subclause shall list the test cases associated with the test design and provide a summary description of each ones.

9 Software unit / integration test case specification

This clause shall provide for identification of software unit test / integration test cases. It shall be organized per identified test case. Each test case can be described through one or several description sheets.

9.m. Test case "m"

This subclause shall provide for definition of the unit / integration test case "m". For the unit / integration test case "m", it shall contain the following subclauses:

9.m.1 Test case identifier

This subclause shall list the test case uniquely. A short description of the test case purpose shall be provided.

9.m.2 Test items

This subclause shall identify the test items. Reference to appropriate documentation shall be performed and traceability information shall be provided.

9.m.3 Inputs specification

This subclause shall describe the inputs required to execute the test case.

9.m.4 Outputs specification

This subclause shall describe the expected outputs

9.m.5 Test pass / fail criteria

This subclause shall list the criteria to decide whether the test has passed or failed.

9.m.6 Environmental needs

This subclause shall identify:
  • the exact configuration and the set-up of the facility used to execute the test case as well as the utilization of any special test equipment (e.g. bus analyser);
  • the configuration of the software utilized to support the test conduction (e.g. identification of the simulation configuration).

9.m.7 Special procedural constrains

This subclause shall describe any special constraints on the used test procedures.

9.m.8 Interfaces dependencies

This subclause shall identify all the test cases that need to be executed before this test case.

9.m.9 Test script

This subclause shall specify or identify all the test script used to execute the test case. The test scripts can be collected in Appendix.

10 Software unit / integration test procedures

This clause shall provide for identification of software unit / integration test procedures. It shall be organized per identified test procedure.

10.p Test procedure "p"

This subclause shall provide for definition of the unit / integration test procedure "p". For the unit / integration test procedure "p", it shall contain the following subclauses:

10.p.1 Test procedures identifier

This subclause shall specify the test procedure uniquely.

10.p.2 Purpose

This subclause shall describe the purpose of this procedure. A reference to each test case used by the test procedure shall be given.

10.p.3 Procedure steps

This subclause shall describe every step of the procedure execution:
  • log: describe any special methods or format for logging the results of test execution, the incidents observed, and any other event pertinent to this test;
  • set up: describe the sequence of actions necessary to set up the procedure execution;
  • start: describe the actions necessary to begin the procedure execution;
  • proceed: describe the actions necessary during the procedure execution
  • test results acquisition: describe how the test measurements is made;
  • shut down: describe the action necessary to suspend testing when interruption is forced by unscheduled events;
  • restart: identify any procedural restart points and describe the actions necessary to restart the procedure at each of these points;
  • wrap up: describe the actions necessary to terminate testing.

Software [Unit / Integration] Test Plan appendices

Appendices shall be included as appropriate, depending on the Software [Unit / Integration] Test Plan to be provided:
  • Test to test cases traceability matrix,
  • Test cases to test traceability matrix,
  • Test scripts,
  • Detailed test procedures.
