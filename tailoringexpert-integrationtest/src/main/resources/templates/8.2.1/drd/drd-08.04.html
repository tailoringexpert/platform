DRD-8.04 - Software Reuse File

Reference: ECSS-E-ST-40C Space engineering - Software
ECSS-Q-ST-80C Space product assurance - Software product assurance

Purpose: To document the results of analysis to be performed on existing SW intended to be reused.

Content:

Title

The document to be created based on this DRD shall be titled "[project] Software Reuse File".

Title page

The title page shall identify the project document identification number, title of the document, date of release and release authority.

Contents list

The content list shall identify the title and location of every clause and major subclause, figure, table and annex contained in the SW Reuse File.

Foreword

A foreword shall be included in the document which describes as many of the following items as are:
  • identification of which organizational entity prepared the SW Reuse File,
  • information regarding the approval of the SW Reuse File,
  • identification of other organizations that contributed to the preparation of the SW Reuse File,
  • a statement of applicability identifying which other documents are cancelled and replaced in whole or in part,
  • a statement of significant technical differences between this SW Reuse File and any previous one,
  • the relationship of the SW Reuse File to other standards or documents.

Introduction

An introduction may be included to provide specific information or commentary about the technical contents.

1 Scope and applicability

This clause shall describe the scope, applicability and purpose of the SW Reuse File.

1.1 Scope

This subclause shall contain a statement to define the goal of the SW Reuse File.

1.2 Applicability

This subclause shall contain a statement to define the use of the SW Reuse File.

2 References

This clause shall contain the following statements:

2.1 Normative references

This subclause shall contain the following statements:
"This document incorporates, by dated or undated reference, provisions from other publications. These normative references are cited at appropriate places in the text and publications are listed hereafter. For dated references, subsequent amendments to, or revisions of, any of these apply to this document only when incorporated in it by amendment or revision. For undated references, the latest edition of the publication referred applies.
[insert document identifier] [insert document title]."

2.2 Informative references

This subclause shall contain the following statement:
"The following documents, although not a part of this SW Reuse File, clarify its contents:
[insert document identifier] [insert document title]."

3 Terms, definitions and abbreviated terms

This clause shall contain the following subclauses:

3.1 Terms and definitions

  1. This subclause shall list any applicable project dictionary or glossary, and all unusual terms or terms with a meaning specific to this Software Reuse File, with the definition for each term.
  2. If a project dictionary or glossary is applicable the following sentence shall be inserted:
    "The definitions of [insert title and identifier of applicable dictionaries or glossaries] apply to this document.
  3. If a list of definitions is required the following sentence shall be inserted:
    "The following terms and definitions are specific to this document:
    [insert term] [insert definition].

3.2 Abbreviated terms

This subclause shall list all abbreviations used in the SW Reuse File with the full meaning or phrase for each abbreviation.

4 Presentation of the software intended to be reused

  1. This clause shall describe the technical and management information available on the SW intended for reuse.
  2. This subclause shall be structured in separate subclauses, one per identified reused SW item.

4.x SW item XY

The following information shall be provided for each SW item:
  • SW item name and main features;
  • developer name;
  • industrial property and exportability constraints if any;
  • applicable disposition for warranty, maintenance, installation and training;
  • commercial SW necessary for software execution if any;
  • development and execution environment (computer, operating system, ...);
  • considered version and list of components;
  • implementation language;
  • size of the SW (e.g. number of source code lines , size of the executable code);
  • Licensing conditions

5 Compatibility of existing SW item with project requirements

The SRF shall describe which part of the project requirements (RB) are intended to be implemented through software reuse. For each of the SW items identified in the previous subclause, a subsubclause denoted as 5.x shall be identified with the following content:

5.x SW item

It shall provide the availability and quality status (completeness, correctness, etc.) of the following information:
  1. software requirements documentation;
  2. software architectural and detailed design documentation;
  3. forward and backward traceability between system requirements;
  4. software requirements, design and code;
  5. unit tests documentation and coverage;
  6. integration tests documentation and coverage;
  7. validation documentation and coverage;
  8. verification reports;
  9. performance (e.g. memory occupation, CPU load);
  10. operational performances;
  11. residual non conformance and waivers;
  12. user operational documentation (e.g. user manual);
  13. code quality (adherence to coding standards, metrics).
For each of the points, the SRF shall document the quality level of the existing software with respect to the applicable project requirements, according to the criticality of the system function implemented.

6 Software reuse analysis conclusions

  1. The SRF shall document the results of the software reuse analysis,
  2. For each software item, the SRF shall provide the following information:
  • decision to reuse or not reuse, based on the information provided in previous chapters;
  • estimated level of reuse;
  • assumptions and methods applied when estimating the level of reuse.

7 Detailed results of evaluation

The SRF shall include the detailed results of the evaluation.

The detailed results of the evaluation can be presented in an appendix.

8 Corrective actions

  1. The SRF shall document any corrective actions identified to ensure that the software intended for reuse meets the applicable project requirements.
  2. The SRF shall document the detailed results of the implementation of the identified corrective actions.

9 Configuration status

The SRF shall include the detailed configuration status of the reused software baseline.
